## High-level data outputs

Primary output roots:

- `data/tie_points_shards/`
  - tie-point + pair + slope metadata shards
  - manifest: `0000_manifest.pkl`
- `data/graph_data_shards/`
  - per-`n` graph data shards
  - manifest: `0000_manifest_p<steps>.pkl`
- `data/tieCuspSlopes.pkl`
  - cusp-only HP sidecar derived from tie slope shards
  - generated by `OBDsaveSourceData.py --save-cusp-data`
- `log/`
  - timestamped CSV run logs for tie pathway

## Pathway 1: tie points + slope metadata

Code path: `save_tie_points(...)` and loaders around `load_tie_points_from_shards(...)`.

### Tie manifest format

Manifest file: usually `data/tie_points_shards/0000_manifest.pkl`

Top-level manifest schema:

- `format`: `"obd.tie_points_slope.shards.v1"`
- `created_at`: timestamp string
- `shards_dir`: absolute path
- `n_entries`: dict keyed by stringified `n`

Each `n_entries[str(n)]` has:

- `n`: `int`
- `shard_path`: relative shard path string
- `tie_points`: `int`
- `local_min`: `int`
- `max_local_min_p`: `float | None`
- `updated_at`: timestamp string

### Shard payload format

Each tie shard payload (`tie_points_nXXXX.pkl`) is a dict:

- `n`: `int`
- `float_by_n`: `np.ndarray` of tie `p` values (`dtype=float64`)
- `float_with_pairs_by_n`: `list[tuple[float, list[tuple[int, int]]]]`
- `tie_slope_by_n`: `list[dict]` (per tie point slope/extremum metadata)

### tie_slope_by_n per-point record (current)

Per record keys:

- `p`: `float`
- `expected_sorted`: `float`
- `slope_left`: `float`
- `slope_right`: `float`
- `dir_change`: `bool`
- `extremum_type`: `"minimum" | "maximum" | "neither"`
- `extremum_ambiguous`: `bool`
- `deadzone_used`: `bool`
- `dir_change_ambiguous`: `bool`
- `quantization_suspect`: `bool` (currently true for dead-zone escalation path)
- `high_precision_used`: `bool`
- `resolution_path`: string stage label (e.g. `"non_deadzone_fast"`, `"deadzone_hp3point"`)
- `resolution_failure_reason`: string reason or `""` when none

### Tie-slope calculation flow (current)

Core function: `_tie_slope_records_for_n(...)`

1. Compute tie points for one `n`.
2. Build per-point local `eps` from nearest tie spacing:
   - `eps = min(slope_a * local_min_gap, slope_eps_cap)` (`local_min_gap` is the smaller of distances to neighboring tie indices, or distances to `(0,1)` endpoints for end ties).
   - non-finite `eps` is replaced by `slope_eps_cap`.
   - then lower-bounded by the **hard minimum** only: `eps = max(eps, slope_eps_min)` (constant `DEFAULT_TIE_SLOPE_EPS_MIN` unless overridden via `--tie-slope-eps-min`).
   - `hit_eps_floor` (return flag and CSV column) means **some** tie had `eps` strictly below `slope_eps_min` *before* that final `max`; i.e. the hard floor actually raised epsilon for at least one point.
3. Float64 baseline 3-point slope:
   - build `p_left = clip(p - eps, 0, 1)` and `p_right = clip(p + eps, 0, 1)`; if either equals `p` (edge-clamp), nudge with `np.nextafter` toward `(0,1)` so the stencil stays one-sided; `stats["ulp_step_adjusted"]` counts those nudges.
   - evaluate `E_sorted` at `[p_left, p, p_right]`
   - compute left/right one-sided slopes
   - classify sign with `slope_flat_tol`
4. Dead-zone trigger:
   - if either side is flat in float64 baseline, escalate to HP 3-point
5. HP 3-point escalation:
   - evaluate same three points at `mpmath` precision (`DEFAULT_HP_DPS`, **40** decimal places in code)
   - recompute slopes
   - classify with tighter HP tolerance:
     - `hp_flat_tol = slope_flat_tol / DEFAULT_TOL_SCALE`
     - `DEFAULT_TOL_SCALE = 1e12`
6. Final status:
   - resolved type: `minimum`, `maximum`, or `neither`
   - ambiguous when signs still flat/non-usable after escalation

CLI (main script `OBDsaveSourceData.py`): `--tie-slope-eps-min` sets `slope_eps_min`. **`--tie-slope-eps-floor-mult` removed** — there is no separate `float64.tiny`-based epsilon floor anymore, and **no ULPS-multiple epsilon floor**.

### Tie CSV log columns

`save_tie_points` CSV includes (column order matches `csv_fields` in code):

- run id: `run_started_at`, `n`
- timing: `compute_sec`, `write_sec`, `iter_total_sec`, `hp_3point_sec`
- counts/summary: `tie_points`, `local_min`, `max_local_min_p`
- flags/counts: `hit_eps_floor`, `deadzone_used`, `deadzone_ambiguous`, `ulp_step_adjusted`
- HP escalation: `high_precision_points`, `hp_3point_attempted`, `hp_3point_resolved`, `hp_3point_failed`, `hp_flat_tol_scale`

**Removed from schema:** `eps_ulps_bumped` (older runs may still have it in historical CSV headers).

## Pathway 2: graph data shards

Code path: `save_graph_data(...)` and `load_graph_data_from_shards(...)`.

### Graph manifest format

Manifest filename embeds p-grid density:

- `0000_manifest_p<steps>.pkl`

Top-level keys:

- `format`: `"obd.graph_data.shards.v2"`
- `created_at`, `updated_at`
- `p_steps`: `int`
- `p_values`: `np.ndarray` (`float32`)
- `n_min`, `n_max`
- `shards_dir`
- `n_entries`

Each `n_entries[str(n)]`:

- `n`
- `shard_path`
- `rows` (= `p_steps`)
- `p_steps`
- `k_count` (= `n+1`)
- dtype descriptors:
  - `dtype_y`: `"float32"`
  - `dtype_perm`: `"uint16"`
  - `dtype_expected_sorted`: `"float32"`
  - `dtype_expected_sorted_slope`: `"float32"`
- `updated_at`

### Graph shard payload format

Each graph shard dict:

- `format`: `"obd.graph_data.n_shard.v2"`
- `n`: `int`
- `p_steps`: `int`
- `y`: `np.ndarray`, shape `(p_steps, n+1)`, `float32`
- `perm`: `np.ndarray`, shape `(p_steps, n+1)`, `uint16`
- `expected_sorted_by_p`: `np.ndarray`, shape `(p_steps,)`, `float32`
- `expected_sorted_slope_by_p`: `np.ndarray`, shape `(p_steps,)`, `float32`

### Graph calculation notes

- PMF is built in float64 using log-domain binomial math (`gammaln`, `log`, `exp`) for stability.
- Stored arrays are cast to compact dtypes (`float32` / `uint16`).
- `expected_sorted_slope_by_p` uses `np.gradient` over the `p_values` grid.

## Pathway 3: cusp sidecar (HP-only)

Code path: `save_cusp_data_from_tie_shards(...)` (`--save-cusp-data`); load with `load_cusp_data(...)` (see [Loader return structures](#loader-return-structures-summary)).

Purpose: store only likely cusp candidates in a compact standalone file so cusp plots/analysis do not need to load all tie-point records.

### Candidate selection

A tie point is included when either condition is true:

- `extremum_type == "minimum"`
- `extremum_ambiguous == True` (potential cusp)

### Center tie at p = 0.5

- `tie_index` is signed relative to the center tie index for that `n`:
  - center point is `tie_index = 0`
- pair-level HP recomputation is intentionally skipped at center (`hp_pair_recompute_skipped=True`)
- HP candidate list is forced to `[0.5]` for this center point

### HP candidate handling

For non-center points, the writer computes HP tie `p` values from each `(i, j)` pair.
With the current non-merged tie pipeline, non-center points normally have one pair, so this is usually a single HP candidate.

### Local `eps` for cusp HP 3-point block

`eps_local_float` on each cusp record comes from `_local_eps_for_tie_index(...)`, which uses the **same** recipe as tie-slope local epsilon: `min(slope_a * local_min_gap, slope_eps_cap)`, then `max(..., slope_eps_min)` (no `float64.tiny` floor, no ULPS floor). Callers pass `slope_eps_min` from the same default / CLI as tie saves.

### File schema: tieCuspSlopes.pkl

Top-level dict keys:

- `format`: `"obd.tie_cusp_slopes.v3"`
- `created_at`, `updated_at`
- `source_tie_manifest`: absolute source tie manifest path
- `hp_dps`: HP precision used for stored strings (same as `DEFAULT_HP_DPS`, **40** in code)
- `include_hp_three_point`: whether optional 3-point HP context block is stored
- `n_entries`: dict keyed by `str(n)`

Each `n_entries[str(n)]`:

- `n`: `int`
- `center_index`: `int` (index in tie array nearest `p=0.5`)
- `center_p_float`: `float`
- `count_candidates`: `int`
- `updated_at`: timestamp string
- `records`: `list[dict]`

Per-cusp record keys:

- identity/context:
  - `n: int`
  - `p_float: float`
  - `tie_index: int`
  - `is_center_tie: bool`
  - `pair_count: int`
  - `pairs: list[tuple[int, int]]`
- inherited classification:
  - `extremum_type: str`
  - `extremum_ambiguous: bool`
  - `resolution_path: str`
  - `resolution_failure_reason: str`
- HP root metadata:
  - `p_hp_main: str`
  - `p_hp_candidate: str`
  - `hp_pair_recompute_skipped: bool`
- HP slope/EV core values (always present):
  - `ev_mid_hp: str`
  - `slope_left_hp: str`
  - `slope_right_hp: str`
- local geometry:
  - `eps_local_float: float`
- optional HP 3-point block:
  - `hp_three_point: dict | None`
  - when present, keys:
    - `p_left_hp`, `p_mid_hp`, `p_right_hp`
    - `ev_left_hp`, `ev_mid_hp`, `ev_right_hp`
    - `slope_left_hp`, `slope_right_hp`
- provenance:
  - `diagnostic_source_hint: "minimum_or_ambiguous"`

## Loader return structures (summary)

### `load_tie_points_from_shards(...)`

Returns:

- `float_by_n: dict[int, np.ndarray]`
- `float_with_pairs_by_n: dict[int, list[tuple[float, list[tuple[int, int]]]]]`
- `tie_slope_by_n: dict[int, list[dict]]`

### `load_graph_data_from_shards(...)`

Returns:

- `format`, `n_min`, `n_max`, `p_steps`, `p_half_start`
- `p_values: np.ndarray(float32)`
- `rows_by_n: dict[int, dict[str, np.ndarray]]` with keys:
  - `y`, `perm`, `expected_sorted_by_p`, `expected_sorted_slope_by_p`
- `manifest_path`

### `load_cusp_data(...)`

Loads `data/tieCuspSlopes.pkl` (or another path). Validates `format == "obd.tie_cusp_slopes.v3"`. Optional `n_list` restricts which `n` blocks are loaded; `require_all` mirrors the tie/graph loaders (missing requested `n` raises).

Returns:

- `format`, `created_at`, `updated_at`, `source_tie_manifest`
- `hp_dps`, `include_hp_three_point`
- `n_entries: dict[int, dict]` — string keys in the file are normalized to `int` keys. Each inner dict matches the per-`n` block written by `save_cusp_data_from_tie_shards` (see [Pathway 3](#pathway-3-cusp-sidecar-hp-only)): `n`, `center_index`, `center_p_float`, `count_candidates`, `updated_at`, `records`.

## Tie diagnostic program data model

Code: `tie_slope_diagnostic.py`

Purpose: visualize and inspect tie slope classification outputs against local 3-point sampling.

### Inputs and internal data

- Recomputes ties via `all_tie_points_float_with_pairs(...)`
- Recomputes slope metadata via `_tie_slope_records_for_n(...)` (must match main pipeline defaults: **no** `slope_eps_floor_mult` / tiny floor / ULPS epsilon floor; only gap scaling + `slope_eps_min`).
- Local diagnostic `eps` arrays mirror the same rule via `_compute_local_eps_arr` (aligned with `_tie_slope_records_for_n`).
- Diagnostic CLI: `--slope-eps-min`, `--slope-a`, `--slope-eps-cap`, `--slope-flat-tol` ( **`--slope-eps-floor-mult` and `--slope-eps-min-ulps` removed** ).
- Selects displayed points by:
  - deadzone/unresolved filter
  - optional manual include (`INCLUDE_TIE_POINT_NUMBERS`)
  - optional reference fallback point near `p=0.5`

### Per-point display data

For each displayed point:

- always computes float64 3-point sample around local `eps`
- if `high_precision_used=True`, also computes HP 3-point sample
- right panel visualizes these actual 3-point sample sets (no quadratic overlay)

### Displayed stage/provenance fields

From slope records:

- `deadzone_used`, `high_precision_used`
- `resolution_path`, `resolution_failure_reason`
- `extremum_type`, `extremum_ambiguous`

Diagnostic-only provenance:

- `diagnostic_source`: one of
  - `deadzone_filter`
  - `manual_include`
  - `reference_fallback`

## Practical interpretation notes

- Final classification semantics are:
  - `minimum`, `maximum`, `neither`, or ambiguous (`extremum_ambiguous=True`)
- `extremum_type="neither"` can occur with `extremum_ambiguous=True`; ambiguity flag is authoritative for unresolved state.
- Timing hierarchy in tie saves:
  - `compute_sec` (total compute)
  - `write_sec` (I/O)
  - `hp_3point_sec` (HP escalation subset of compute)
